# はじめに

データエンジニアを10年ほど続けてきました。データ基盤は一度作って終わりではなく、運用していくなかで初めて見えてくる落とし穴がたくさんあります。きれいな設計論はいくつもの場所で語られていますが、実際に手を動かしてきた人間として、私自身が **本当に痛い目に遭った失敗** と、そこから何を学んだかを共有するほうが、これから基盤を作る方の役に立つのではないかと思いました。

本記事では、私のキャリアのなかで特に糧になった失敗事例を2つ取り上げ、それぞれから得た教訓をまとめます。

そして、それらすべての根っこにあるのは、たった一つの最重要事項です。

> **要件定義を徹底する。**

要件を疎かにせず、しっかり聞き出すこと。これが何より大切です。最近では私自身もクライアント先に出向き、直接要件を伺うワークスタイルをとっていますが、このひと手間を惜しまないかどうかで、出来上がるデータ基盤の品質はまったく変わってきます。

それでは、具体的な失敗事例から見ていきましょう。

# 失敗1: 過去履歴のスナップショットが保持されていなかった

## 何が起きたか

最初に作ったデータ基盤は、**常に最新のデータだけを保持・生成する仕組み** でした。ソースから取ってきたデータでテーブルを上書きし、参照側はその最新値を見にいく、というシンプルな構造です。

ある日、夜間バッチが途中で失敗していました。朝のうちに気づければよかったのですが、検知が遅れ、昼すぎから夕方にかけて再実行する判断になりました。

ところが、再実行した時点ではすでにソース側の値が更新されていたのです。結果、

- 顧客が見たいのは「当日朝時点のデータ」
- しかし基盤に残っていたのは「夕方再実行時点のデータ」
- **朝時点のスナップショットはどこにも存在しない**

という事態になりました。データの欠損というより、**時間軸が壊れた** という、地味ですが取り返しのつかない事故です。

## 教訓 —— 「どの時点のデータを見るか」を要件として握る

この失敗の本質は、技術的な話の前に **要件定義の漏れ** にあります。

顧客が見たいのは「いま現在のデータ」なのか、「ある時点のデータ」なのか、あるいは「時系列での変化」なのか。同じ「データを見たい」という言葉でも、求める時間軸はまったく違います。

この経験以降、私は要件ヒアリングで次のような問いを必ず立てるようになりました。

- そのデータは **どの時点の状態** が必要なのか（最新／日次の朝時点／任意の過去時点）
- 過去に遡って分析する必要があるか
- 値が更新されたとき、**前の値も残しておく** 必要があるか

そして、これらの要件が「ある」と分かった時点で、**履歴を保持する設計** をシステムに組み込みます。具体的には、いわゆる **SCD Type 2**（Slowly Changing Dimension Type 2）のような、変更前後の値を時系列に積んでいくテーブル設計です。

履歴テーブルがあれば、ジョブが失敗してリランで値が変わっても「朝時点のスナップショット」は残ります。再現性が担保され、何時時点のデータがどう使われたかが追えるようになります。**履歴保持は、起きうる事故の数だけ価値が増していく投資** だと考えるべきです。

# 失敗2: タイムスケジュール連結のパイプラインが崩壊した

## 何が起きたか

別のプロジェクトで、こんなパイプラインを組んでいました。

- **処理A**: 毎時5分に起動
- **処理B**: 処理Aの完了を見込んで毎時45分に起動

「処理Aは通常30分くらいで終わるから、45分なら余裕でしょ」という、いま振り返るとゾッとする時間ベースの連結です。

当初は問題なく動いていました。しかしある日、処理Aが遅延・失敗するようになると、後続の処理Bが **未完了または不完全なデータ** に対して走るようになります。すると、

- 不完全なデータで集計が回り、それが下流に流れる
- 「どの時点のデータがどこに送られたか」が誰にも追えなくなる
- 一見は成功しているので、しばらく異常に気づけない

という最悪のパターンに陥り、多くの障害を引き起こしました。

## なぜそんな設計にしたのか

弁明させてください（笑）。これには背景があります。

もともと日次バッチで動いていた基盤に、後から「**もっとリアルタイムでデータが欲しい**」という顧客要件が乗ってきたのです。既存の仕組みを大きく作り変える時間と予算はなく、**既存処理の上にラップアップする形** で、毎時のスケジュールを継ぎ足してしまいました。

短期的には動いて見えるのですが、それが上記の脆さを生みました。

## 教訓 —— タイムスケジュール連結は禁じ手

この失敗から得た一番大きな教訓は、**タイムスケジュールによる処理の連結はデータ基盤の禁じ手** だということです。

「前のジョブが時間内に終わる前提」を時刻で表現してしまうと、その前提が崩れた瞬間にデータの整合性も崩れます。しかも、失敗ではなく「成功したように見える誤り」を生むため、検知が極めて困難です。

代わりに採るべき方式は、次のいずれかです。

- **トリガー式（イベント駆動）**: 前段の処理完了を検知して後続を起動する
- **ファイル検知式**: 所定の場所にファイルが置かれたことを検知して自動更新する

最初のジョブの起動はスケジュールでよいのですが、**そこから先は必ず「前工程の完了」をトリガーにする**。Databricks Workflows、Airflow、Dagster などのモダンなオーケストレーターは、すべてこの思想で作られています。

「スケジュールを数珠つなぎにしているパイプライン」を見たら、迷わずリファクタリング対象だと判断してください。

# 失敗3: リアルタイム要件を鵜呑みにした

失敗2の話には続きがあります。「リアルタイムで欲しい」と言われて作った仕組みは、技術的な構造の問題だけでなく、**そもそもリアルタイムが必要だったのか** というレベルでも誤りでした。

## 鵜呑みのコスト

顧客から「リアルタイムでデータを見たい」と言われたとき、私は要件を深掘りせずにそのまま実装に入りました。その結果、

- ストリーミング処理は **インフラコストが高く**、常時リソースを食い続ける
- 設計が複雑になり、**ジョブが失敗したときの再実行が極めて難しい**
- なぜか **たまにログが欠損する** という、原因特定に時間を吸われる問題が発生する

という、技術的にも運用的にもしんどい状態が続きました。

## 深掘りしてわかった本当の要件

落ち着いて顧客に「どのタイミングで、誰が、何のためにデータを見るのか」を聞き直したところ、本当に必要だったのは **数時間ごとのバッチ連携** で十分だったことが判明しました。

しかも顧客にとって譲れなかったのは、リアルタイム性ではなく「**処理が一件でも合わないことは許容できない**」という確実性のほうだったのです。

つまり、

- 求められていたもの: **データの正確性・確実性**
- 提供してしまったもの: **(欠損の可能性のある) リアルタイム性**

という、価値の方向性がそもそもズレた実装になっていました。確実性が最優先の場面では、無理にリアルタイム処理を導入するべきではありません。

## 教訓 —— 「リアルタイム」は深掘りしてから判断する

「リアルタイムで欲しい」という言葉は、データ基盤において最も警戒すべきフレーズの一つです。多くの場合、その裏にある本当のニーズは違うところにあります。

- **「最新の状況をすぐ把握したい」** → 数分〜1時間のバッチで十分なことが多い
- **「いつでも正確な値を取りたい」** → 鮮度ではなく **正確性** が本当の要件
- **「異常をすぐ検知したい」** → アラート設計の話であって、データ鮮度の話ではない

リアルタイム化の判断をする前に、必ず次を確認します。

1. 本当にそのデータを **何分後・何時間後** に見るのか
2. データが遅延した場合、**実害が出るのはどのタイミング** か
3. 鮮度と確実性、どちらが優先か（多くの業務は確実性が勝る）
4. リアルタイム化のコスト（インフラ・実装・運用）を顧客と合意できるか

そして、もしリアルタイム化を本当に採用するなら、**リスクを把握したうえで、許容するデータ品質のレベルを顧客と明確に合意** する必要があります。「リアルタイムでExactly Onceで欠損ゼロ」は技術的にも運用的にも非常に重いコミットメントであり、軽々しく約束していい話ではありません。

# まとめ —— すべては要件定義に戻ってくる

3つの失敗を振り返ると、根っこにある原因はどれも同じでした。

- **失敗1（履歴喪失）**: 「どの時点のデータが必要か」という要件を聞けていなかった
- **失敗2（タイムスケジュール連結）**: 「リアルタイム要件」を後付けで雑に乗せ、設計の前提を聞き直さなかった
- **失敗3（リアルタイム鵜呑み）**: 「リアルタイム」という言葉の裏側を深掘りせず、本当の要件である「確実性」を取り違えた

データエンジニアリングの仕事は、技術的な巧拙の前に、**「何を作るべきか」を正しく握れているか** で勝敗が決まります。要件をしっかり聞き出すこと、そしてその要件が時間軸・品質・確実性のどこに重みを置いているのかを見極めること。これができれば、技術的な選択肢は自然と絞れてきます。

最近、私自身がクライアント先に出向いて直接要件を伺うワークスタイルをとっているのも、まさにこの「要件定義を徹底する」ためです。フロアの空気、業務のリズム、本当に困っている瞬間を見ないと拾えない要件があります。

これからデータ基盤を作る方、運用している方の参考になれば幸いです。失敗は宝です。一緒に良い基盤を育てていきましょう。
