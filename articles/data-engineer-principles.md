# はじめに

## いろいろなキャリアを渡ってデータエンジニアに流れ着いた話

私は最初からデータエンジニアだったわけではありません。

- **大学院時代**: 機械学習のモデル作成。SVM、カーネル法、ベイズなど、いまでは古のと呼ばれる手法をゴリゴリ書いていました

- **ベンチャー**: フロント・バックエンドでプロダクト開発。データプロダクトチームを立ち上げ、EM（Engineering Manager）も経験
- **その後**: 全社横断のデータ基盤チームを立ち上げ、データマネジメントとデータ基盤構築

データには常に関わっていましたが、機械学習・プロダクト開発・マネジメントを経て、ぐるっと回ってデータエンジニアに行き着いた、という変わったキャリアです。

## 失敗談を共有したい

データエンジニアを10年やっていると、運用してから初めて気づく落とし穴がたくさんあります。きれいな設計論はネットに溢れていますが、**実際に痛い目に遭った失敗** のほうが学びになると思い、本記事を書きました。

そのすべての根っこにあるのは一つです。

> **要件定義を徹底する。**

最近は私自身、クライアント先に出向いて直接要件を伺うようにしています。このひと手間で基盤の品質はまるで変わります。

# 失敗1: 過去履歴のスナップショットを保持していなかった

最初に作った基盤は、最新データだけを上書き保持する仕組みでした。

ある日、夜間バッチが失敗。検知が遅れて夕方に再実行したところ、**ソース側の値が変わっていて朝時点のデータが二度と取れなくなりました**。顧客が見たかったのは「朝時点」だったので、データの欠損というより時間軸が壊れた事故です。

## 教訓

「どの時点のデータが必要か」という要件を聞けていなかったのが本質です。

ヒアリングでは次を必ず確認します。

- 必要なのは最新／特定時点／時系列のどれか
- 値が更新されたとき、前の値も残す必要があるか

履歴保持が必要なら、**SCD Type 2** のような設計をシステムに組み込みます。事故が起きたときに価値が跳ね上がる投資です。

# 失敗2: タイムスケジュール連結のパイプラインが崩壊した

別のプロジェクトで、こんなパイプラインを組んでいました。

- **処理A**: 毎時5分に起動
- **処理B**: 毎時45分に起動（Aが終わっている前提）

Aが遅延・失敗した瞬間に、Bが不完全なデータを処理し始めます。**失敗ではなく「成功したように見える誤り」が下流に流れ続け**、検知も困難という最悪のパターンに陥りました。

背景は、日次バッチの上に「リアルタイムでデータが欲しい」という要望をラップアップで継ぎ足したことでした。

## 教訓

**タイムスケジュールによる処理連結は禁じ手** です。代わりに以下のいずれかにします。

- **トリガー式**: 前段の完了を検知して後続を起動
- **ファイル検知式**: 所定の場所にファイルが置かれたら更新

最初の起動だけはスケジュールでよいですが、その先は必ず前工程の完了をトリガーにします。スケジュールが数珠つなぎになっているパイプラインを見たら、即リファクタ対象です。

# 失敗3: リアルタイム要件を鵜呑みにした

失敗2の続きです。そもそも「リアルタイム」が本当に必要だったのか、という根本的な誤りでした。

「リアルタイムで欲しい」と言われ深掘りせず実装した結果、

- ストリーミングのインフラコストが高い
- 障害時の再実行が極めて難しい
- なぜかログが欠損する

という地獄を見ました。落ち着いて顧客に聞き直したら、本当に必要だったのは **数時間バッチ** で、譲れないのは「**一件でも値が合わないのは許容できない**」という確実性でした。

つまり、求められていたのは **正確性**、提供したのは **欠損リスクのあるリアルタイム性**。価値の方向がそもそもズレていました。

## 教訓

「リアルタイムで欲しい」は最も警戒すべきフレーズです。実装前に次を確認します。

- 本当に何分・何時間後に見るのか
- 遅延の実害はどのタイミングで出るのか
- 鮮度と確実性、どちらが優先か（多くは確実性が勝つ）
- リアルタイム化のコスト（インフラ・実装・運用）を顧客と合意できるか

採用するなら、許容するデータ品質レベルを **顧客と明確に合意** します。「リアルタイムでExactly Onceで欠損ゼロ」は軽々しく約束していい話ではありません。

# まとめ —— すべては要件定義に戻ってくる

3つの失敗の根は同じでした。

- **履歴喪失**: 「どの時点」を聞けていなかった
- **タイムスケジュール連結**: 後付け要件を雑に乗せた
- **リアルタイム鵜呑み**: 言葉の裏の本当の要件（確実性）を取り違えた

データエンジニアリングの勝敗は、技術より先に **「何を作るべきかを握れているか」** で決まります。

## IVRyはこの観点で恵まれた環境

宣伝になってしまいますが、IVRyのデータチームはここで戦うのにとても良い環境だと感じています。

- **検索やNLPに詳しいメンバーがいる**: コンテキストエンジニアリングを学ぶうえで、隣で議論しながら進められる
- **顧客の商談に同席するのが当たり前**: FDE的な動き方が文化として根付いていて、要件を現場で直接握れる

「データエンジニアとしてAI時代に必要な力を伸ばしたい」と思っている方には、本当におすすめです。一緒に働ける方をお待ちしています。

これからデータ基盤を作る方、運用している方の参考になれば幸いです。失敗は宝です。
